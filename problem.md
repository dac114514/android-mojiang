# 墨匠 Rewrite UI 设计问题分析

> 基于对完整项目（17 个 Kotlin 源文件，全 Jetpack Compose + Material 3）的 UI 审查

---

## 1. 导航系统问题

### 1.1 底部标签栏非悬浮设计
- 当前使用标准 Material 3 `NavigationBar`，紧贴屏幕底部，与背景融为一体
- 88% 透明度 + 零高度阴影，在视觉上缺乏"浮层"的层次感
- 与内容区域的过渡生硬，没有明确的视觉边界

### 1.2 页面可访问性断裂
- `AppDestination` 定义了 7 个路由，但 `bottomDestinations` 仅包含 5 个
- **PromptsScreen（提示词）** 和 **SettingsScreen（设置）** 没有任何导航入口
- 用户无法通过正常交互到达这两个页面，属于导航系统的严重缺陷
- PromptsScreen 的交互实际耦合在 RewriteScreen 的模板选择中

### 1.3 图标不一致
- `Icons.Rounded.Psychology` 同时被用于 Models 和 Prompts 两个不同的目的地
- 底部栏的图标选择缺乏语义一致性

---

## 2. 视觉设计问题

### 2.1 配色使用不充分
- `Color.kt` 定义了完整的暗/亮色板（主色 `#6E4FF6`，强调色 `#FF7D5C`，成功色 `#20C997` 等），但实际 UI 中很少使用
- 几乎所有 Card 都使用 `surface` / `surfaceVariant`，页面呈现大面积灰/白色块
- 强调色（橙色）仅在 GradientHeroCard 中出现，其他地方几乎看不到
- 成功/警告色完全没有在 UI 中使用

### 2.2 卡片样式单一，缺乏层次
```kotlin
// 所有屏幕都在重复这个模式
Card(colors = CardDefaults.cardColors(containerColor = MaterialTheme.colorScheme.surface))
```
- 所有卡片都是 0dp 阴影、相同的圆角、相同的背景色
- 主卡片、次要卡片、操作卡片之间没有任何视觉权重区分
- 信息层级被完全扁平化，用户无法通过视觉快速判断内容重要性

### 2.3 排版层级扁平
- `SectionTitle` 中的副标题文字过长（如"真实读取本地项目、章节、模型和导出状态"），信息噪音大
- 标题/副标题/内容的字体大小差距过小
- 正文和辅助信息在视觉上难以区分

---

## 3. 具体页面问题

### 3.1 DashboardScreen（概览页）
- 统计卡片（StatCard）样式完全一致：灰色背景 + 左文右数布局 × 4
- "快捷入口"部分（FeatureRow）看起来可点击，但实际并未绑定导航
- 信息密度过低：4 个卡片 + 4 行就占满了一屏
- 缺乏实时数据可视化元素（进度条、占比图等）

### 3.2 ProjectScreen（项目页）
- **章节内容查看使用 `AlertDialog`**：
  - 弹窗内展示长文本存在严重的滚动体验问题
  - 无法查看上下文，关闭后回到列表顶部
  - 原文/加料后切换通过小型 `OutlinedButton` 实现，交互面积小
- 章节列表每项包含 3 个按钮（看原文/看加料后/清空结果），操作密度过高
- "自动分章"按钮行为令人困惑（仅设置一行状态文本而不实际处理）
- 当前小说卡片和章节列表之间缺少视觉分隔

### 3.3 RewriteScreen（加料页）
- **超长单屏表单**：处理范围 + 改写策略 + 提示词模板 + 自定义编辑器 + 保存模板 + 操作按钮 + 状态 + 队列信息全部堆叠
- 多个主要操作按钮互相竞争焦点：
  - "后台处理范围"、"真实改写首章"、"保存模板"、"重试失败"、"清理完成"
- 加料强度使用普通 `Slider`，无刻度/标签辅助
- 状态文本混杂在按钮之间（`LocalNovelStore.statusMessage.value`）
- "长篇一致性记忆"卡片内容单薄，用途不明确

### 3.4 ModelsScreen（模型配置页）
- **信息密度极高**：配置选择器 + 编辑表单 + 操作按钮 + 可用模型网格 + 已保存配置列表 全在一屏
- 表单字段过多（配置名称、提供商名称、Base URL、当前模型、API Key），缺少分组
- 可用模型以纯文本标签展示在网格中，触摸目标不明确
- 编辑区与列表区的地板不明显

### 3.5 ExportScreen（导出页）
- 可用功能单一，但使用了与复杂页面相同的布局模板
- "导出范围"和"导出当前章"功能类似，用户可能困惑该用哪个
- 导出记录列表缺少"分享/打开文件"操作

### 3.6 SettingsScreen（设置页）
- 深色模式开关 `darkMode` 是局部 `remember { mutableStateOf(true) }`，**实际上不生效**
- 所有设置项堆在一个大 Card 中，没有任何分组标签
- 数据摘要以纯文本 `\n` 分隔展示，没有任何格式

### 3.7 PromptsScreen（提示词页）
- 无导航入口可到达此页面
- 缺少搜索/筛选功能

---

## 4. 交互与反馈问题

### 4.1 缺少操作反馈机制
- 没有任何屏幕使用 `Snackbar` 或 `Toast`
- 所有状态反馈通过 `LocalNovelStore.statusMessage.value` 以纯文本展示
- 文本状态的视觉样式（成功/错误/警告）无法区分

### 4.2 加载状态无视觉指示
- 网络操作（测试连接、拉取模型）没有 `CircularProgressIndicator`
- 后台改写队列的进度没有直观的进度条
- 用户无法感知操作正在执行

### 4.3 章节范围选择体验差
- 使用两个纯文本 `OutlinedTextField` 输入数字，无防错引导
- 缺少可视化范围选择（如滑块选择器）

---

## 5. 架构与可维护性问题

### 5.1 状态管理
- 所有状态集中在 `LocalNovelStore` 单例中，UI 直接读写可变状态
- 状态变更通知通过 `mutableStateOf` 的 Compose 快照系统隐式传递
- 无 `ViewModel` 层意味着状态在配置变更（屏幕旋转）时会丢失

### 5.2 无公共组件体系
- 7 个屏幕各自实现了类似的 `SectionTitle` + `Card` 布局模式
- 没有统一的列表项组件
- 没有统一的空状态组件
- 没有统一的错误展示组件

---

## 问题总结

| 类别 | 严重程度 | 问题数 |
|------|---------|--------|
| 导航缺陷 | 高 | 2 |
| 视觉层次缺失 | 高 | 4 |
| 交互反馈不足 | 高 | 3 |
| 页面设计缺陷 | 中 | 10 |
| 可维护性 | 中 | 2 |

**核心矛盾**：应用的功能逻辑（小说改写工作台）是完整且经过设计的，但视觉呈现和交互体验没有匹配其功能复杂度，导致用户感知到的品质远低于实际功能价值。
